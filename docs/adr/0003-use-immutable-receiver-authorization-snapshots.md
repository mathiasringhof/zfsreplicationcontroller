# Use immutable receiver authorization snapshots

The receiver activates each complete set of receive permissions as one immutable Receiver Authorization Snapshot, with every accepted identity bound to the policy from that same snapshot. Each grant is identified by the exact Kubernetes object incarnation rather than its reusable namespace/name, preventing a deleted and recreated task from inheriting an earlier task's authority. Invalid tasks are rejected as individual grants rather than blocking publication for unrelated valid tasks; only system-wide observation or publication failures retain the previous snapshot. Snapshot changes are driven by Kubernetes events and local expiry deadlines rather than periodic rebuilding, while locally enforced grant expiry bounds stale authority. Expiry and revocation close admission at both SSH authentication and forced-command start, but do not cancel a receive command already admitted. A Receive Task is reported Ready only after its exact grant is active on the identified ready receiver Pod; status remains an eventually consistent observation and live SSH admission is authoritative. Snapshot generation remains receiver-local rather than becoming part of the public status contract. This favors continuity for valid replications without exposing partial authorization state, allowing an unavailable control plane to extend a grant indefinitely, or interrupting an active receive merely because its admission window closed.

Each snapshot is materialized as a complete immutable generation in a private directory. A stable `authorized_keys` manifest, atomically replaced from that generation and carrying its snapshot identity in a strict ignored header, is the sole publication point used by SSHD. Every forced command carries both the snapshot and grant identities needed to resolve its exact policy. Incomplete staged generations are unreachable; retired generations are cleanup candidates, while active-snapshot admission checks revoke their authority independently of cleanup and commands that already loaded a plan continue.

The active-manifest rename is the irreversible authority commit point. All authority-critical validation and writes occur before it and leave the previous manifest active on failure; after it succeeds, activation is reported successful, rollback is forbidden, and cleanup failures are warning-level retryable housekeeping.

Command admission requires the referenced snapshot to still be active, not merely present on disk, and rechecks the active reference immediately before returning a plan. That check is the admission instant: a concurrent activation either leaves the operation admitted under the old snapshot or denies it for retry, while delayed or failed retired-generation cleanup cannot preserve authority.

Snapshot identity is the full SHA-256 digest of a canonical ordering and encoding of every compiled grant, while grant identity derives from the exact task UID. Identical desired authority is therefore a no-op, and every authority-bearing change produces a different opaque, shell-safe snapshot identity independent of Kubernetes list ordering.

Snapshots are a derived runtime cache, not durable authority. A receiver process starts with no active authorization, obtains a fresh Kubernetes list, activates an initial snapshot, and only then becomes ready and starts serving SSH; it does not recover a generation left by a previous process. Last-complete retention therefore applies only to transient failures within one running receiver process.

A synchronized view with no eligible grants activates the canonical empty snapshot and is a successful initial state. The receiver may serve SSH and become Kubernetes-ready with no authorized identities; idle health is distinct from failure to establish authoritative state.

The Receiver remains long-lived node-level infrastructure rather than being created per Replication Run. Completion or failure makes that run's Receive Task terminal and removes only its grant in the next snapshot; it does not make the shared Receiver unhealthy or end its lifecycle.

After initial activation, failure to publish a newer candidate leaves the Receiver ready but degraded while the prior active snapshot remains readable and enforceable. New grants are not reported Ready and publication is retried; loss of trust in the active snapshot makes the Receiver unready or terminates it.

An SSH identity is unique within a snapshot. If multiple otherwise-eligible tasks contain the same canonical public key, every conflicting grant is rejected and the unrelated grants are still publishable; snapshot compilation never merges policies or chooses a winner by ordering.

Snapshot grants use a closed private representation compiled from the Receive Task rather than serializing the public policy type directly. Every capability is explicit, all authority-bearing fields are validated before publication, and malformed or unknown persisted data is rejected; legacy missing-field normalization is removed. The private format is not versioned because generations do not survive process restart and all binaries ship as one release.

This refactor preserves the existing Syncoid/ZFS command language, capability meanings, dataset boundaries, pipeline and batch limits, redirects, and executable allowlist. It adds no policy capability or CRD field; behavior changes are limited to enforcing the new snapshot invariants, preserving explicit denial values, and making the existing expiry field a renewable lease.

## Verification

Tests use the module interface with temporary directories to cover compilation, identity, expiry, atomic activation, filesystem hardening, admission, and the existing command policy; redundant tests of the replaced shallow decomposition are removed. Kubernetes-adapter tests cover initial synchronization, complete-view events, status isolation, Pod readiness, and lease renewal. One focused real-cluster test exercises actual OpenSSH authentication before and after expiry, terminal revocation, unrelated-grant continuity, and node-level Receiver readiness.

One Receiver Authorization module owns the complete authority decision from API-neutral grant candidates through validation, deterministic snapshot construction, atomic activation, exact grant resolution, expiry and revocation admission, and command authorization into an opaque executable plan. Kubernetes observation and status updates remain in a Kubernetes adapter, while process execution remains in an execution adapter; neither adapter can inspect raw persisted policies or reproduce authorization rules.

The module presents a minimal synchronous interface: the receiver daemon replaces authority from a complete candidate set and receives only activation outcomes, while a forced-command process submits an invocation and receives only an opaque authorized plan. Event-loop ownership, Kubernetes status effects, and process execution remain outside this interface; pure compilation and filesystem seams are private implementation details used by the module's own tests.

Concretely, a publisher is constructed from a runtime root and receiver rules, then accepts complete grant-candidate sets and returns only accepted task UIDs, isolated rejection messages, change state, and the next lease deadline. A separate admission operation accepts only opaque snapshot/grant references plus the original SSH command and returns an opaque executable plan whose standard streams are supplied by the forced-command adapter. Snapshot layout, policy values, authorized arguments, clocks, and failure-injection seams do not cross the external interface.

The Kubernetes adapter waits for an initially synchronized informer cache and submits a complete candidate view after each relevant task event or local expiry. It never feeds incremental mutations into the module or infers deletion from an incomplete watch stream; watch recovery remains the adapter's concern, and content-derived identity makes unchanged complete views no-ops.

The adapter uses controller-runtime's shared informer cache and maps all relevant task events to one coalesced receiver-authorization reconciliation. Each reconciliation lists the complete node-local view; deadline requeues drive lease transitions, and an explicit initial trigger activates even an empty snapshot. Controller-runtime owns cache synchronization, watch reconnection, backoff, and event coalescing rather than a custom watch state machine.

Activation is the authority transition; Kubernetes status is an asynchronously retried report of its result and never rolls an activated snapshot backward. Ready and rejection outcomes are applied only when the current task UID matches the result, terminal tasks are not reopened, and status failure leaves local authority unchanged. Before creating a sender Job, the run controller verifies that the exact receiver Pod UID reported by the task still exists and is currently Kubernetes-ready.

Per-grant rejection is an expected isolated activation outcome carrying the exact task UID and a concise safe operator message; it does not block unrelated grants and introduces no stable public rejection-code contract. System-wide failures before the manifest commit return an activation error and retain prior authority, while post-commit housekeeping failures are warnings.

Each generated authorized-key entry carries OpenSSH's native expiry option as well as its restricted forced command, with the timestamp rounded conservatively so authentication cannot extend past grant expiry. Command admission independently checks the exact expiry. A local expiry event still publishes the next snapshot and updates task status, but a delayed event cannot extend either authentication or command admission.

Receive Task expiry is a renewable Authorization Lease rather than a maximum run duration. The run controller may move only `expiresAt` monotonically forward while the run is nonterminal; task identity, key, destination, and capabilities remain immutable. Renewal produces a new snapshot identity, already-admitted commands continue, and later SSH connections use the renewed generation. Failure to renew eventually expires the grant locally even during control-plane loss.

The lease horizon is 30 minutes and the run controller renews it back to that horizon when 10 minutes remain, using an explicit scheduled reconciliation rather than incidental activity. These timings remain internal policy rather than public configuration; the margin accommodates ordinary clock skew, event delay, and transient API failure while bounding stale authority to approximately 20–30 minutes after renewal stops.

The authorization module rejects a candidate whose observed expiry is more than 35 minutes in the receiver's future: the 30-minute policy horizon plus at most five minutes of clock-skew tolerance. Receiver-side validation is authoritative, so manually created tasks cannot bypass the bounded lease by supplying a distant deadline.

Lease lapse is terminal for that Receive Task UID within the observing receiver-process lifetime and permanently once Failed status is persisted. The run controller does not renew a lease it observes as lapsed, and a command admitted before lapse may still finish but cannot admit follow-up commands afterward. After receiver restart, fresh nonterminal Kubernetes state is authoritative, so a valid renewal durably recorded before a partition but missed by the old receiver process may restore the grant; no local generation is recovered.
